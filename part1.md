## 개념 정리
* AI Agent: 주어진 목표를 달성하기 위해 스스로 판단하고 행동하는 인공지능 시스템

* Zero Data Retention(=ZDR): 데이터를 보관하지 않는 것. 
  * 학습하는 것과 데이터를 보관하는 것은 다르다. 학습은 데이터 유출의 가능성이 있고, 데이터 보관은 해킹 또는 내부 직원으로 인한 리스크가 존재한다.

## AI 사용은 위험할까?
> 네이버, 카카오, 당근이 OpenClaw 금지했대. 왜 쓰면 안되는데?

우선 짚고 넘어가야할 부분은 OpenClaw는 오픈소스이며, 공식적인 제품이 아닙니다.

OpenClaw 자체가 AI 모델이 아니기 떄문에, 시중에 나와있는 AI 모델 중 원하는 제품을 선택하여 사용할 수 있습니다
사용자가 설정한 목표를 수행하기 위한 질문(AI API 기반)을 하면, AI 모델이 판단해서 답변이 내려지고, open claw 는 그대로 수행합니다.

* AI API는:
  * “변호사가 법률 자문을 해주는 것”
* OpenClaw는:
  * “그 자문을 그대로 실행하는 집행관”

→ 즉 아무리 위험한 명령이 오더라도 이를 막을 안전 장치가 없다는 게 문제입니다.

상용 프로그램은 아래와 같이 답변을 제한적인 권한과 통제 하에서 수행합니다.
* 위험 명령 차단
* 사용자 승인 필요
* 실행 범위 제한
* 반복 횟수 제한

## Claude Code도 취약점 발견됐다던데? 위험한 거 아니야?
* 발견된 3개 모두 오픈소스 사용 또는 내부 직원의 악의적인 공격으로 인해 발생할 수 있는 위험에 대한 취약점이었습니다.
* 이로 인해 실제 피해 사례는 없었으며 사용자들로 인해 발견된 문제가 아니라, Check Point Research 가 취약점을 발견한 후 Anthropic과 협력하여 패치된 것입니다.
* 위와 같이 취약점을 패치했다는 것만으로 OpenClaw와의 차이점을 알 수 있습니다. 연구 기관과 협력하여 취약점을 발견하고, 검증 과정을 추가하여 사용자가 안전하게 사용할 수 있는 환경으로 개선하고 있습니다.
* 지금까지도 VS Code 나 Intellij의 확장 플로그인, npm 오타를 노린 악성 패키지, 악의적인 코드를 숨긴 오픈소스 프로젝트는 존재했으며, AI 사용 시에 발생할 수 있는 동일한 종류의 취약점들은 지녔습니다.
* AI 도구가 특별히 위험한 문제를 가지고 있는 것보다, “자동화 범위가 넓어진만큼 설정 파일 하나의 영향력이 커졌다” 는 게 핵심입니다.


## 그럼 어떻게 써야해?
* 오픈소스 사용 제한
  * 꼭 필요한 경우, 신뢰도가 높은 오픈소스에 한하여 부문 검증 후 사용하는 것이 좋을 것으로 보입니다.
* MCP 사용 제한
  * MCP 도 대부분 오픈소스이기 때문에 위 오픈소스 항목과 동일하게 대응하는 것이 좋을 것 같습니다.
* 공식 AI 서비스 선택
  * 신뢰도가 높은 ChatGPT, Claude, Gemini와 같은 공식 제품을 사용하여 검증 및 보안 과정을 거치도록 하는 게 좋습니다.
* 피해 범위 격리
  * 홈 디렉토리 작업 X. 운영 계정 외부 MCP 연결 X(필요 시 검토)
  * 최소 권한 IAM 사용
* 신규 기능 검증
  * AI 에이전트를 사용한다고 하더라도 신규 기능 사용 전에는 미리 검토하는 게 필요해보입니다.

## 결론
* Claude code는 Openclaw 보다 cursor 와 비슷한 선상에서 비교해야합니다.
* 현재 H.AI에서도 AI를 사용하고 있고, cursor도 사용하는데 claude code가 안된다는 건 보안의 해결책이 되지 않는다고 생각합니다.
* 보안을 위해서라면, claude cowork, codex, cursor, Antigravity와 같은 다른 AI 에이전트의 툴을 함께 검토하는 게 좋아보입니다.
* 특정 AI 서비스를 막는다고 해도 비슷한 AI 서비스들은 매우 많습니다. “Claude code 사용 금지” 같은 방향보다 “신규 AI 서비스 사용 전에는 검토 필요“와 같은 AI 사용 규칙을 수립하는 게 더 좋다고 생각합니다.
* AI 에이전트 사용을 하지 않는다고 해도 잘못된 블로그 정보, 오타를 노린 악성 패키지 공격, 내부 직원의 공격, 해킹 등의 문제는 여전히 존재합니다.

실수는 AI 에이전트 사용 여부와 무관하게 항상 발생할 수 있는 영역인데,
AI가 정교해지면서 코드 리뷰와 디버깅을 통해 실수를 예방하고, 바로 잡는 데 효율적으로 사용할 수 있기 때문에 트레이드오프라고 생각합니다.

결론적으로 AI 에이전트 사용은 막지 말되, AWS IAM 과 같은 계정 권한 제한, 오픈소스는 사용 전 검토를 통해 통제하고,
관리자는 사용자들이 AI 를 어떻게 사용하는지 관리할 수 있도록 비즈니스 플랜을 사용하는 것이 좋다고 생각합니다.

## AI 대상 공격
### H.AI와 같은 챗봇 서비스 대상
* 챗봇에 들어온 요청의 악의적인 공격에 대한 검증 필요
* 프롬프트 무효화 공격에 대한 고려 필요
* 민감 데이터 연결 X
### 공격 유형
* 프롬프트 인젝션: 	
  AI에게 악의적인 지시를 주입해 의도치 않은 동작을 유발하는 공격
* AI 메모리 포이즈닝 (Memory Poisoning): AI 어시스턴트의 장기 메모리(Workspace/Memory/Knowledge 등)에 악의적 지시나 허위 정보를 주입해, 이후 대화·추천에 지속적으로 영향을 미치게 하는 공격
### AI 에이전트의 도움을 받는 개발자 대상
* 로컬 AI 에이전트는 일반 챗봇보다 더 많은 액션이 가능함 → 외부 콘텐츠를 AI에게 소스로 주는 것 주의
* 오픈소스 프로젝트 사용 주의
### 공격 유형
* 프롬프트 인젝션
